April 7, 2014 (5.5 hours):
Now that I2C functionality has been wrapped nicely into SW functions, some design was undertaken with respect to the I2C LED driver/controller. Group dimming and frequency control, turning off the auto-increment feature, and ignoring all pattern control and and I2C subaddresses were all design decisions made today. The first test will be with OE LED state register set to be entirely on, ignoring all dimming, frequency, etc. Unfortunately, the LED driver will not be testable until a protoboard can be obtained and populated.
Further investigation into the motor settings was made. The hope was to use the built in Enhanced PWM's full-bridge mode, but it was noted that the pins in the current design do not make up a full peripheral pinout. There are also some doubts as to whether the full-bridge mode is the correct output for full stepper motor functionality. The current solution is to flywire the rest of the PWM peripheral to the existing micro traces.

April 8, 2014 (4.5 hours):

Meaningful data was observed on the scope, but output on the LED driver pins did not follow any meaningful pattern. The I2C communication was slowed to ensure that it met the driver's specified speed constraints.

April 8, 2014 (3 hours):
Further I2C inspection revealed that the driver failed to nack the requests from the microcontroller. Some small changes to the software yielded successful mitigation of the nacking problems. Unfortunately, the output remained unchanged. The I2C resistors and supply voltage were modified a bit to ensure that an I2C 'high' signal was indeed being delivered, but the successful acking of the address byte implies that this threshold issue is not the underlying problem here.

April 9, 2014 (5 hours):
MAN LAB session was held. LED driver I2C was (still) debugged. The driver continued to ack requests from the microcontroller, but output continued to fail to follow any meaningful request pattern. The decision was made to assemble the part on the board and retest it in the near future.

April 10, 2014 (4.5 hours):
A switch in focus was made to the motor. The output of the full-bridge PWM module was, upon inspection, deemed to be be unfit with respect to the motor's specifications. Instead, a GAL device was obtained and a step sequence was programmed into it, with a simple clock signal originating from a micro pin. This worked nicely, but a further improvement was made by ignoring the GAL device altogether and simple toggling the sequence on GPIO's, initiated by a timer interrupt. This solution uses the existing micro traces and does not require the addition of a GAL device, which is ideal.

Weekly Work Total: 15 hours
Project Work Total: 196 hours

April 27, 2014 (5 hours) SPI interactions between The microcontroller and Raspberry Pi were modified to seek and relay input from the webapp. The Raspberry Pi communicates and undertakes the next-song action, which prepares the micro for metadata consumption. The Raspberry Pi then sends 4 20-char strings with meaningful metadata, which the microcontroller stores for subsequent transmission to the LCD via I2C.

Week 16

April 28, 2014 (6 hours) Attempting to start pulling sub-components together an run them simultaneously were undertaken. Specifically, incorporating LED driver updates and ADC sampling with the SPI/webapp/LCD portion was the main integration effort. Unfortunately, a malfunction between the micro and Raspberry Pi resulted in an inoperable Raspberry Pi. A replacement was promptly ordered.

April 29, 2014 (12 hours) The motor circuitry was re-worked; some de-soldering had loosened a connection on a level-translator pin. Pushbuttons were momentarily inoperable, but the electrical issue was resolved once all had been packages (floating pins). A buffer chip: 74HC244N was included to isolate the microcontroller and Raspberry Pi. Packaging and integration efforts continued with mixed success. All the PSSCs were demonstrable, but the simultaneous powering of more than 3-4 individual LEDs created inexplicable noise on the SPI line, which broke the rest of the functionality.

May 1, 2014 (3.5 hours) The root cause of the integration issue was discovered: the current draw on the 5V supply created some instability in the buffer chip, which had been incorrectly biased with 5V (rather than the SPI logic level 3.3V). Once 3.3V was brought to the buffer chip, the device operated correctly.

May 2, 2014 (6 hours) The IP configuration was modified heavily to allow for maximum interaction at the Design showcase. A bridge to a laptop LAN port was made for hosting; unfortunately, this does not allow users to access the webapp via Wi-Fi. This was an unfortunate restriction of the MSEE Atrium demo space (no network drops were available).
The device was later demo'ed at the Design Showcase.
